How Bugs Are Born: A Model to Identify How Bugs Are Introduced in Software Components

Abstract

To identify the origin of software bugs, many studies assume that "a bug was introduced by the lines of code that were modified to fix it'' (or variations of it). However, this assumption may not hold because the location identified as the origin of the bug, in fact, is not where the bug was first introduced, e.g., a bug introduced in an external API. The lack of empirical evidence on when bugs are introduced causes that researchers cannot understand under what circumstances this assumption is true or false.

With the intention of better understanding the origins of bugs, this paper explores in detail how 116 bugs were introduced in two different software projects. From this observation, we propose a model for determining the first time that a software exhibits bugs. This model uses several assumptions for deciding whether, after a change, a bug is present in the software. Then, we operationalize the model to create curated datasets that allow the evaluation of the real performance of four algorithms when identifying the origin of bugs. Our results show that there were bugs introduced without changing the source code of the project in a 9\-21\ of the analyzed cases. The F-score of the algorithms varies from 0.42 to 0.66, and the percentage of false positives was noteworthy (>50\).

Raw Data

Nova

Nova is a project from OpenStack and it is mainly written in Python. It uses Gerrit as code review system and Launchpad as issue tracking system. The data used in this work was gathered from January 2015 to December 2015, we gathered 7563 issues that fulfills with the requirements (bugs, closed and commited). We randomly selected 60 issues of the initial 1801 issues that can be found here.

ElasticSearch

ElasticSearch is written in Java. It uses Github Pull Request as code review system. The data used in this work was gathered from January 2013 to December 2016, we gathered 1801 issues that fulfills with the requirements (bugs, closed and commited). We randomly selected 60 issues of the initial 1801 issues that can be found here.

Analysis

To ensure that the bug reports can be applied to our model, we verified that they describe real bug reports at the moment of their report and not other issues. For that, we carefully read the description and comments in the issue tracking system and code review system to analyze whether we can apply them the model.

After this analysis, we removed 3 bug reports from the initial set of 60 random bug reports in Nova and 1 bug report from the initial set of random bug reports of ElasticSearch since it was not clear that they describe bugs at the moment of the report. The data can be found here.

During this analysis we used some scripts that can be found here.

Curated reference datasets

Then, 57 bug reports from Nova and 59 form ElasticSearch were manually analyzed in order to identify the BIC, when it was possible, and how they are related to the BFC. This manual analysis was carried our keeping in mind the proposed model. After that, we have the curated reference datasets, that can be the "ground truth" for each project. This "ground truth" was build with the BFCs whose BICs certainly were manually identify, and the BFCs that does not have BICs for sure.

After these exclusion steps, the datasets contain 57 bug reports from Nova and 59 from ElasticSearch linked to their respectives BFC. These datasets can be found curated reference datasets.
